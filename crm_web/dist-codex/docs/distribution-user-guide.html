SEEDCRM DISTRIBUTION SCHEME B

分销本系统使用说明

本文面向 SeedCRM 后台使用人员，说明外部分销成交数据进入本系统后，如何查看会员、处理预约排档、完成门店履约、查看回推结果和处理异常。

适用版本：方案 B V1 最后更新：2026-05-01 文档格式：HTML

1. 使用边界

分销方案 B 的定位是：外部分销系统负责商品、小程序、会员注册、分销关系、分佣、提现和外部资金流；SeedCRM 只承接已支付成交数据、预约排档、门店履约和履约状态回推。
关键红线：分销已支付订单不进入 Clue。SeedCRM 不处理外部分销佣金、提现、退款资金流。异常重试由 DistributionEventIngestService.replayFromScheduler 可信重放。禁止直接删除 Customer / Order / PlanOrder 数据。
事项 SeedCRM 内处理 外部分销系统处理
已支付订单入站 处理 创建或匹配 Customer，创建 Order(paid) 推送 distribution.order.paid 事件
线索 / 客资池 不处理 分销已支付订单不进入 Clue 外部成交前的推广、浏览、下单由外部系统负责
预约排档与门店履约 处理 进入 PlanOrder，完成服务后回推 接收 SeedCRM 的履约完成状态
佣金、提现、外部分销退款资金流 不处理 仅记录必要状态和内部绩效影响 负责计算分佣、提现审核、外部资金退款
禁止把 ExternalDistributionPaidOrder 当成新的核心模型；它只是入站事件。禁止绕过 Customer / Order / PlanOrder 建立平行履约链路。

2. 业务主链路

1. 外部已支付 外部分销系统完成会员购买，推送已支付成交事件。
2. Customer 系统按手机号或外部会员身份创建 / 匹配会员客户。
3. Order(paid) 创建已支付订单，记录外部订单号、交易号、原始报文。
4. PlanOrder 客服预约排档，门店核销和服务确认后进入履约。
5. Outbox 回推 服务完成后异步回推 crm.order.used 给外部分销系统。 
ExternalDistributionPaidOrder -> Customer -> Order(paid) -> PlanOrder -> Order(used) -> Outbox(crm.order.used)

标准抖音或普通客资仍遵循 Clue -> Customer -> Order -> PlanOrder。分销已支付成交是例外入口，但进入系统后仍必须回到 Customer -> Order -> PlanOrder 主链。

3. 角色与权限

角色 主要职责 可操作范围
PRIVATE_DOMAIN_SERVICE 查看会员信息、客户画像、标签与企微触达 默认只能查看授权范围内会员；原始报文和敏感字段脱敏
STORE_SERVICE 门店订单核销、服务确认、服务完成 按门店数据范围查看和处理订单，不配置接口
INTEGRATION_OPERATOR 查看接口日志、异常队列、回推结果、重试结果 只能看结果和脱敏诊断，不可修改核心业务数据，不可执行全局核心重放
INTEGRATION_ADMIN 配置分销接口、签名密钥、状态映射、调度策略 可执行全局队列处理、异常重试、回查对账和完整诊断
ADMIN 系统管理、权限配置、风险确认 拥有全局权限，需要负责上线前风险确认和数据治理审批
PARTNER_APP 外部分销系统或合作应用访问自身数据 仅限自身 partner_code 范围，不能触发全局批处理

权限同时受 RBAC 和 ABAC 控制。敏感字段包括手机号、金额、原始报文、签名、token、防重复编号、外部会员 ID、外部交易号等，非授权角色默认脱敏。

4. 页面入口

私域客服 -> 会员信息

这是 Customer 视图，不是新模型。使用“分销来源 / 抖音来源”切换，分销来源展示外部分销已支付成交客户。

注意 会员信息页不能直接创建 Customer、Order，也不能触发分佣、提现、退款资金处理。

客资中心 -> 顾客排档

已支付订单可以进入预约排档。客服根据门店档期安排客户到店，排档后进入门店服务流程。

门店服务 -> 订单列表

门店人员处理核销、服务确认单、服务中、订单完成等动作。完成后本地订单进入已履约状态。

系统设置 -> 调度中心 -> 分销接口

集成角色查看接入配置、履约回推队列、异常队列、回查对账结果、幂等健康和验收样本。

系统设置 -> 基础配置 -> 域名配置

管理员配置系统基础域名和 API 域名。分销入站、回查、对账、Swagger/OpenAPI 和回调地址都从这里生成，避免上线后仍显示本地地址。

分销接口 -> 生产联调信息

首屏可以复制入站地址、查看使用说明、查看联调说明、打开 Swagger 接口定义，并进入联调工作台做 dry-run 验证。

5. 日常操作流程

5.1 查看分销成交会员

入口 私域客服 -> 会员信息 -> 分销来源
适用场景 确认外部分销已支付客户是否进入 SeedCRM，查看最近订单、来源、外部身份、标签和企微绑定。
结果判断 能看到 source=distribution 的会员和关联订单，说明已支付入站链路已生效。

5.2 预约排档

入口 客资中心 -> 顾客排档
适用场景 客服对已支付订单安排门店档期，或修改 / 取消已预约档期。
结果判断 订单与门店档期关联后，后续门店服务可以基于订单进入核销与服务确认。

5.3 门店履约与服务完成

入口 门店服务 -> 订单列表 -> 服务确认单 / 查看服务单
适用场景 门店人员核销线上团购券或定金，填写服务确认单，客户签字确认，进入服务中并最终完成订单。
结果判断 PlanOrder 完成后，本地 Order 进入已履约，系统写入 scheduler_outbox_event 等待异步回推。

6. 队列与异常处理

6.1 履约回推队列

状态 含义 处理建议
待推送 PENDING 已生成履约回推事件，等待调度处理。 一般等待自动调度；管理员可手动处理队列。
推送中 PROCESSING 调度器正在调用外部分销系统。 稍后刷新查看结果。
成功 SUCCESS 外部分销系统已收到履约完成回推。 无需处理，可复制追踪编号核对。
失败 FAILED 回推失败但仍可重试。 检查回推地址、签名密钥、外部接口响应后重新入队。
死信 DEAD_LETTER 多次失败后停止自动重试。 需管理员排查配置或外部接口，再手动重新入队。

6.2 异常队列

异常队列用于处理入站失败、重复报文冲突、外部订单冲突、状态冲突、签名失败、防重复编号缺失等问题。

发现失败 查看异常类型、外部订单号、本地订单号、追踪编号。
核对冲突 查看冲突字段、失败原因、处理建议。
修正数据 修正外部报文、接口配置或状态映射。
重新入队 管理员触发异常重试，系统可信重放入站服务。
确认闭环 成功后变为已处理；无需处理时标记处理。
异常状态 说明 下一步
OPEN 待人工核对。 修正外部数据或配置后重新入队，或确认无需处理后标记处理。
RETRY_QUEUED 已重新入队，等待调度器可信重放。 等待处理完成；失败会回到 OPEN。
HANDLED 已处理或人工确认无需处理。 保留记录用于审计。
异常重试只能通过调度器重新调用 DistributionEventIngestService.replayFromScheduler，禁止人工绕过服务直接写 Customer / Order / PlanOrder。

6.3 回查对账结果

状态回查和对账拉取用于补偿外部分销订单取消、退款、对账失败等情况。若需要改变本地状态，调度器必须生成 distribution.order.* 事件并重放入站服务，不直接更新核心表。

7. 防重复健康

分销入站依赖 X-Idempotency-Key、eventId 和外部订单唯一键防止重复创建订单。页面上“防重复编号”对应接口联调中的幂等键。

检查项 正常结果 风险处理
唯一索引 幂等唯一索引已生效。 若存在历史重复接收日志，索引暂未生效，上线前需完成数据治理或风险确认。
重复分组 重复分组为 0。 按页面“建议保留、需核对记录、治理口径、处理建议”执行人工治理。
日志样例 可查看接收记录、追踪编号、处理状态、内容指纹、关联订单。 非授权角色看到脱敏内容；完整诊断仅管理员和集成管理员可见。
历史重复日志治理是只读诊断和人工确认流程，不提供一键删除。禁止直接删除 Customer / Order / PlanOrder 数据。

8. 验收清单

  • 分销已支付订单不进入 Clue。
  • 入站后创建或匹配 Customer，并创建 Order(paid)。
  • 会员信息页能在分销来源看到会员和最近订单。
  • 订单可预约排档并进入 PlanOrder。
  • 门店完成服务后写入履约回推 Outbox。
  • 回推成功、失败、死信均可在分销接口页查看。
  • 异常 OPEN、重新入队、已处理状态可闭环。
  • 回查 / 对账失败可以定位追踪编号和处理建议。
  • 幂等健康能显示重复分组和治理建议。
  • 操作员敏感字段脱敏，管理员可查看完整诊断。
  • 域名配置保存后，分销接口页展示的入站地址、状态回查地址、对账地址和 Swagger 地址同步更新。
  • Swagger/OpenAPI 只对管理员、集成管理员和集成操作员开放，不能对公网未授权访问。
  • 真实模式保存前已补齐 API 域名、签名密钥、回推目标、状态回查路径和对账路径。

9. 常见问题

为什么分销成交不进入 Clue？

Clue 表示未成交或待转化客资。分销入站的数据已经在外部系统完成付款，所以直接进入 Customer + Order(paid)，不参与公海、分配、线索转化率统计。

为什么 SeedCRM 不做分佣和提现？

方案 B 选择由外部分销 / 电商系统负责完整分销体系和资金流，SeedCRM 只负责门店履约和状态回推，避免两个系统重复计算佣金和资金状态。

回推失败会不会回滚门店服务完成？

不会。履约完成先在本地事务内落库，再通过 Outbox 异步回推外部系统。外部接口失败只影响回推队列状态，不回滚本地履约。

看到唯一索引未生效怎么办？

先导出并核对重复接收日志，按建议保留记录和需核对记录进行人工治理。治理完成后由管理员或 DBA 重建唯一索引，期间禁止直接删除业务主链数据。
