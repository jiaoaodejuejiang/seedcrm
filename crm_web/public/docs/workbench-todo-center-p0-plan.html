SeedCRM P0-6 工作台与今日待办中心详细落地方案

工作台与今日待办中心是角色化操作入口，不是经营大屏，也不是新的业务流程引擎。它只聚合、排序、提醒和跳转，把用户带回原模块完成受控操作。

P0-6 Workbench 今日待办 角色入口 RBAC + ABAC 不新增核心业务模型

1. 目标与边界

  • 第一屏只回答一个问题：当前登录角色今天最该处理什么。
  • 标准主链路仍然是 Clue -> Customer -> Order -> PlanOrder。
  • 分销 B 方案仍然是 ExternalDistributionPaidOrder -> Customer -> Order(paid) -> PlanOrder。
  • 工作台只能从 Clue、Customer、Order、PlanOrder、Scheduler、Permission、Salary 等源模块读取待办和风险。
  • 工作台不直接创建、修改、删除 Customer、Order、PlanOrder，也不直接推进主链路状态。
  • 工作台不处理外部分销佣金、提现、退款资金流；这些仍由外部分销系统负责。
  • 待办状态不是业务真实状态，真实状态仍来自原业务对象。
  • 所有入口必须复用原模块 RBAC + ABAC，不能出现“原页面无权限，工作台有权限”的越权。
  • 统计数字只服务于行动排序，不作为页面主视觉；工作台不是堆统计卡片的看板。

数据约束

  • todo_key 必须稳定，例如 CLUE:123:FOLLOW_UP、OUTBOX:456:RETRY。
  • workbench_todo_snapshot 只能写读侧快照，不得反写源业务表。
  • status_code 为稳定编码，status_label 来自字典中心或展示快照。
  • payload_digest_json 只能保存脱敏摘要，不保存 raw_data、token、secret、auth_code、签名明文。
  • 忽略待办只更新 workbench_read_state，不改变源业务对象状态。
  • 工作台缓存 key 必须包含 userId、roleCode、dataScope、storeId、partnerCode、permissionVersion。

4. 接口定义

接口层要围绕“概览、待办、风险、入口、偏好、审计”组织。工作台接口原则上只读；少量写操作只能写偏好、阅读状态和审计。

统一工作台

GET /workbench/overview
GET /workbench/summary
GET /workbench/todos
GET /workbench/todos/counts
GET /workbench/todos/{todoKey}
GET /workbench/risks
GET /workbench/actions
  • /overview 返回当前角色第一屏需要的紧急处理、今日待办、需要关注、最近完成和环境提示。
  • /todos 支持 sourceModule、todoType、priority、status、dateRange、page、pageSize。
  • /todos/counts 计数必须使用与列表一致的权限过滤，不能泄露无权限数量。
  • /risks 聚合超时未跟进、预约临近、Outbox 失败、异常队列、配置待发布、权限冲突等风险。
  • /actions 根据当前用户 RBAC、菜单配置和数据范围返回快捷入口。

入口与偏好

GET  /workbench/action-entry/{todoKey}
POST /workbench/todos/{todoKey}/dismiss
GET  /workbench/preferences
POST /workbench/preferences
  • /action-entry/{todoKey} 返回目标路由、动作权限、阻断原因和风险提示，不直接执行业务写操作。
  • /dismiss 只能忽略提醒，必须填写原因，不能改变原业务状态。
  • /preferences 保存个人固定模块、隐藏类型、默认筛选，不影响其他用户。

调度风险入口

GET /workbench/scheduler-alerts
GET /workbench/integration-todos
GET /workbench/salary-todos
GET /workbench/permission-todos
  • Scheduler 待办只显示 Outbox、异常队列、调度失败、Provider token/auth_code 过期、上线检查失败等入口。
  • 异常重放、真实重试、状态回查、对账拉取仍跳转调度与队列中心执行。
  • Salary 待办只聚合 SeedCRM 内部薪酬确认、结算提醒，不处理外部分销佣金、提现、退款资金流。
  • Permission 待办仅 admin 可见，用于权限草稿待发布、角色菜单冲突、策略预览失败提醒。

5. 权限规则

工作台必须先按权限过滤，再聚合展示。不能先全量查出再让前端隐藏。

ABAC 数据范围

  • self：当前用户是负责人、绑定客户负责人、薪酬本人。
  • team：资源负责人在当前用户团队内。
  • store：当前门店等于订单或服务单归属门店。
  • all：仅授权管理角色可用。
  • partner：当前 partnerCode 等于资源 partnerCode。

字段脱敏

  • 手机号默认展示为 138****1234。
  • 金额展示继续遵循 amount.visibility.* 配置。
  • raw_data、异常 payload、token、secret、auth_code、签名默认不在卡片中展示。
  • traceId、idempotency_key、providerCode、callback log 放入“技术详情”折叠区。
  • 完整敏感信息查看需要单独权限、查看原因和审计。

6. 流程

6.1 工作台加载流程

  1. 用户登录后解析 PermissionRequestContext。
  2. 加载当前角色、菜单权限、按钮权限、数据范围、门店、团队、partnerCode、权限版本。
  3. 根据角色加载工作台布局配置和待办规则。
  4. 按模块执行聚合查询或读取 workbench_todo_snapshot。
  5. 每个模块先做 SQL 级 ABAC 过滤，再做权限守卫二次校验。
  6. 使用字典中心将稳定编码翻译为中文标签。
  7. 对手机号、金额、raw_data、外部身份、异常 payload 做脱敏。
  8. 按优先级、到期时间、等待时长、模块权重排序。
  9. 返回紧急处理、今日要做、需要关注、我发起的、最近完成等分组。
  10. 写入访问审计，审计内容仅保留脱敏摘要。

6.2 待办点击流程

  1. 用户点击待办主按钮。
  2. 调用 /workbench/action-entry/{todoKey} 校验权限和数据范围。
  3. 接口返回目标路由、可用按钮、阻断原因和风险提示。
  4. 前端跳转原业务页面或打开轻量抽屉。
  5. 真实业务写操作由原模块受控接口完成。
  6. 原业务处理完成后，工作台刷新时待办自然消失或更新状态。

6.3 忽略提醒流程

  1. 用户点击忽略提醒。
  2. 系统要求填写原因。
  3. 后端只写 workbench_read_state.dismissed_at 和审计。
  4. 不改变源业务状态，不关闭异常队列，不更新订单或服务单。

6.4 分销 B 方案待办流程

  1. 外部分销成交数据通过 Open API 或 Scheduler 入站服务进入 Customer + Order(paid)。
  2. 工作台可为门店、客服、集成管理员生成预约、核销、异常、回推相关待办。
  3. 分销 paid order 不进入 Clue 待办、公海、分配、回收、线索转化率。
  4. 取消、退款中、已退款事件只针对已有订单生成提醒或集成异常，不创建新订单。
  5. 履约完成后 Outbox 失败显示在集成角色工作台，处理仍跳转调度与队列中心。

6.5 高风险入口流程

  1. 工作台只展示风险和入口，不直接执行高风险动作。
  2. 批量重试、死信重新入队、真实调度执行、配置发布、切换 LIVE、权限变更必须跳转对应治理页面。
  3. 改档、取消预约、退款提醒等业务动作必须进入原模块抽屉确认，展示对象、当前状态、下一步结果和原因输入。
  4. 查看敏感信息必须记录原因和审计。

7. 页面体验设计

第一屏

  • 最多展示三个主分组：紧急处理、今日要做、需要关注。
  • 统计数字只作为分组角标，不再铺满顶部。
  • 每条待办只保留一个主按钮，次级动作放入更多菜单。

待办卡片

  • 展示事项标题、为什么出现、当前状态、等待时长、风险标签和下一步动作。
  • 技术字段默认折叠，脱敏信息默认展示。
  • 空状态必须提示下一步，例如“今天暂无待办，可查看我的客户”。

桌面端布局

  • 顶部显示角色问候、环境模式、今日主任务摘要。
  • 左侧主区域放行动队列，右侧放预警、快捷入口、我发起的、最近完成。
  • 复杂处理跳转原页面，不在工作台塞完整配置表单。

移动和平板

  • 移动端使用单列任务流，复杂表格改为卡片。
  • 平板端使用左待办、右详情分栏，适合门店现场操作。
  • 高风险操作在移动端增加确认或要求管理员桌面端处理。

10. 验收标准与禁止行为

验收标准

  • 不同角色登录后第一屏展示不同待办内容，且与 RBAC / ABAC 权限一致。
  • 第一屏主区域是待处理事项，不是统计卡片堆叠。
  • 每条待办都能说明事项、原因、状态、下一步主操作。
  • 工作台只聚合和引导，不承载复杂配置、复杂排档、复杂异常治理。
  • 待办点击后进入原模块或轻量抽屉，真实业务操作仍由原模块受控接口完成。
  • 分销 paid order 不出现在 Clue 待办、公海、分配、回收、线索转化率中。
  • Outbox、异常队列、状态回查、对账差异能进入集成角色工作台。
  • 未授权数据不能出现在 counts、列表、详情、导出中。
  • 手机号、金额、raw_data、外部会员 ID、外部订单号按权限脱敏展示。
  • 忽略提醒不改变原业务状态。
  • 状态中文来自字典或展示转换，英文编码不反写成中文。
  • 操作完成后提示下一步，例如“已确认到店，下一步等待服务完成”。

禁止行为

  • 禁止新增独立 Task / Workflow 作为核心业务模型驱动主流程。
  • 禁止在工作台直接创建 Customer、Order、PlanOrder。
  • 禁止让分销 paid order 进入 Clue 待办。
