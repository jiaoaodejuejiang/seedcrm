SeedCRM P0-8 系统需求配置化平台详细落地方案

系统需求配置化平台建议在产品侧命名为“配置化平台”或“配置治理中心”。它不是万能低代码后台，而是一条受控配置生产线，把反复需要改代码的系统能力沉淀为可草稿、可校验、可预览、可发布、可回滚、可审计的配置能力。

P0-8 配置治理 草稿 / 发布 / 回滚 权限预览 Dry-run 不落库 不新增核心业务模型

1. 目标与边界

  • 目标是把菜单、角色、权限、字典、页面展示、表单模板、列表字段、按钮动作、待办规则、Provider、Scheduler、规则试算等能力统一收敛到配置生产线。
  • 标准客资链路仍为 Clue -> Customer -> Order -> PlanOrder。
  • 分销 B 链路仍为 ExternalDistributionPaidOrder -> Customer -> Order(paid) -> PlanOrder。
  • 配置平台只能改变系统能力、页面呈现、入口、规则试算和受控参数，不能创建新的核心业务模型。
  • 配置不能直接写 Customer、Order、PlanOrder，也不能让 PlanOrder 脱离 Order 独立存在。
  • 配置不能让分销已支付订单进入 Clue，也不能把外部分销佣金、提现、退款资金流放到 SeedCRM 内部处理。
  • Open API 和 Scheduler 仍必须复用统一入站服务，Scheduler 不得通过配置直写核心表。
  • 所有业务写操作仍由原业务模块服务执行，配置平台只负责生成安全、可审计、可回滚的运行时配置。
  • 普通业务人员只使用发布后的业务页面，不进入配置平台后台。

4. 接口定义

接口统一挂在 /config-platform。所有写操作都先进入草稿，发布前必须完成校验、权限预览和影响预览。

接口用途权限
GET /config-platform/workspaces查询配置空间和环境。config:view
GET /config-platform/packages按配置域、模块、状态、风险等级查询配置包。config:view
GET /config-platform/packages/{packageCode}查询配置包详情、草稿、当前发布版本和最近审计。config:view
POST /config-platform/packages/create-draft从当前发布版本或模板创建草稿。config:draft
POST /config-platform/items/save-draft保存配置项草稿，不影响运行时。config:draft
POST /config-platform/validate校验 schema、字段引用、权限引用、主链路边界和安全门禁。config:draft
POST /config-platform/dry-run规则、Provider、Scheduler、待办、按钮动作等试算，不落核心业务表。config:debug
POST /config-platform/preview-diff展示草稿与当前版本差异。config:view
POST /config-platform/preview-permission按角色或用户预览菜单、按钮、字段、数据范围和脱敏效果。config:view
POST /config-platform/preview-impact发布前预览影响模块、影响角色、影响门店、风险项和阻断项。config:view
POST /config-platform/publish发布配置版本，写目标配置表、发布记录、审计并失效缓存。config:publish
POST /config-platform/rollback-preview基于历史版本生成回滚预览。config:view
POST /config-platform/rollback以新版本方式回滚，不覆盖历史。config:publish
GET /config-platform/versions查询配置版本历史。config:view
GET /config-platform/audit-logs查询配置审计，敏感值脱敏。config:audit
GET /config-platform/runtime/{moduleCode}业务页面读取运行时配置。原模块 view 权限 + ABAC
GET /config-platform/schema/{pageCode}读取页面、列表、字段、按钮、表单 schema。原页面权限 + ABAC

5. 权限规则

RBAC

  • admin：配置平台全量查看、草稿、校验、发布、回滚、审计。
  • integration_admin：Provider、Callback、Open API、Scheduler、接口调试、Outbox、异常队列相关配置；生产发布建议需要管理员复核。
  • integration_operator：只读、dry-run、联调、日志、健康检查，不可发布高风险配置，不可切换 LIVE。
  • clue_manager：可维护线索分配、线索列表、工作台待办等低/中风险配置草稿，不可发布全局权限和核心状态配置。
  • store_service：只读取发布后的门店服务页面配置，不进入配置平台后台。
  • finance：只读取或维护受控薪酬展示/结算配置，不可配置外部分销资金流。
  • private_domain_service：只读取会员、标签、企微相关发布配置，不可维护平台级密钥。
  • partner_app：不进入后台配置平台，只能访问授权的 Open API 文档、回调联调和自身 partner 范围接口。

ABAC

  • 所有运行时配置解析必须带 PermissionRequestContext，并按 self / team / store / all / partner 数据范围过滤。
  • 配置 scope 可支持 GLOBAL / STORE / ROLE / USER / PARTNER，但前端参数不能提升数据范围。
  • 配置预览必须支持按角色和按用户两种方式，叠加门店、团队、partner、字段脱敏。
  • 发布前必须提示权限扩大风险，例如普通角色获得全量订单导出、金额完整查看、raw_data 查看、LIVE 操作等。
  • 敏感字段如 raw_data、手机号、密钥、token、auth_code、签名、支付流水、外部订单原文必须在接口、审计、diff、日志中脱敏。

强制门禁

  • 配置规则不得包含任意 SQL、Groovy、JavaScript 或可执行脚本。
  • 按钮动作只能从白名单选择：跳转、打开抽屉、提交已有受控接口、dry-run、受控重试、导出。
  • 动作配置不得直接调用 Mapper 写 Customer、Order、PlanOrder。
  • Provider 入站仍必须走统一入站服务，Scheduler replay 仍必须走可信重放服务。
  • cancelled、refund_pending、refunded 事件不能通过配置创建新订单。
  • 生产发布、LIVE 切换、权限扩大、字段脱敏放开、批量导出、重试回放必须二次确认并写审计。

6. 流程

配置生命周期

  1. 选择配置域：菜单权限、字典参数、页面列表、表单模板、按钮动作、工作台待办、Provider、Scheduler、规则。
  2. 选择业务对象：Clue、Customer、Order、PlanOrder、Scheduler、Permission、Salary。
  3. 从当前发布版本或模板库创建草稿。
  4. 在单一职责页面编辑配置项，例如列表配置只处理列、筛选、排序、操作列。
  5. 保存草稿，草稿不影响线上运行。
  6. 执行 schema 校验、字段引用校验、权限引用校验、主链路边界校验。
  7. 执行 dry-run 或运行时预览，确认不会落核心表、不会调用真实外部 LIVE。
  8. 执行权限预览和影响预览，展示角色、用户、门店、模块、敏感字段、数据范围变化。
  9. 高风险配置填写发布原因并二次确认。
  10. 发布配置版本，事务内写目标配置表、版本、发布记录和审计。
  11. 按模块精准失效缓存，业务页面读取新运行时配置。
  12. 出现异常时从发布记录进入回滚，生成回滚草稿，预览差异后发布为新版本。

运行时解析

  1. Controller 解析登录态，生成权限上下文。
  2. 读取当前已发布配置版本。
  3. 按模块加载菜单、字典、页面、列表、字段、按钮、规则。
  4. 执行 RBAC 和 ABAC，得到最终可见页面、按钮、字段和数据范围。
  5. 执行字段脱敏、状态中文化、按钮可用性判断。
  6. 返回页面 schema 和业务数据，业务写操作仍回到对应模块服务。

7. 产品与用户体验方案

配置工作台

首页只展示正在编辑的草稿、待校验配置、待发布配置、最近发布版本、高风险配置和可复用模板，不展示大段说明文字。

配置向导

从业务目标开始，例如新增列表视图、调整表单字段、新增按钮动作、修改 Provider 映射；默认从模板库复制，不建议空白创建。

设计器布局

左侧字段/组件库，中间业务预览画布，右侧属性面板，底部固定保存草稿、校验、预览、提交发布。

权限预览

支持按角色、用户、门店、partner 预览最终菜单、按钮、字段、数据范围和脱敏效果。

发布中心

发布前展示差异、影响范围、风险等级、阻断项；回滚必须从发布记录生成回滚草稿。

错误提示

错误使用业务语言，例如“按钮未绑定权限，发布后无法控制谁能点击”，并提供修复入口。

9. 测试与验收

  • 配置草稿不影响线上运行。
  • 配置发布成功后，目标配置、版本、发布记录、审计一致。
  • 配置发布失败时整体回滚。
  • 回滚以新版本发布，不覆盖历史版本。
  • 菜单权限、接口权限、按钮权限、字段权限、ABAC 结果一致。
  • 普通业务角色不能进入配置平台后台。
  • 权限扩大、导出开放、raw_data 查看、LIVE 切换、重试回放必须高风险确认。
  • dry-run 不写核心业务表，不调用真实外部 LIVE。
  • 字典中文化不反写业务状态编码。
  • 列表和字段配置不能突破数据范围和脱敏策略。
  • 服务单模板不能创建独立 PlanOrder。
  • 按钮动作不能直接写 Customer、Order、PlanOrder。
  • Provider Mapping 改动必须经过 dry-run 和 validate。
  • Scheduler 配置改动不能绕过 job lock、confirm-run、Outbox、异常队列。
  • 分销 paid order 在任何配置下都不能进入 Clue。
  • cancelled、refund_pending、refunded 在任何配置下都不能创建新 Order。
  • 敏感字段在接口、日志、审计、diff、影响报告中均脱敏。
  • 配置导入必须校验 schema、权限、版本、环境和高风险项。

10. 禁止行为

  • 禁止把配置平台做成可新增核心业务模型的低代码建模平台。
  • 禁止允许业务人员配置数据库表、任意 SQL、任意脚本、任意接口调用。
  • 禁止配置改变 Clue -> Customer -> Order -> PlanOrder 主链路。
  • 禁止配置让分销 paid order 进入 Clue。
  • 禁止配置让外部系统直接创建 Customer、Order、PlanOrder。
  • 禁止配置让 Scheduler 直接写核心业务表。
  • 禁止配置让 PlanOrder 不绑定 Order 独立存在。
  • 禁止按钮配置绕过原业务接口和后端权限。
  • 禁止把核心状态机变成业务人员可随意配置的状态流。
