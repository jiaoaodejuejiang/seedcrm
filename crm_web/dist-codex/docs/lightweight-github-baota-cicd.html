SeedCRM 轻量 CI/CD 到宝塔服务器部署说明

推荐流程：本地安装依赖并完整测试，确认没问题后推送 GitHub；GitHub Actions 构建前后端 Docker 镜像并推送镜像仓库；宝塔服务器通过 SSH 拉取新镜像并重启容器。

本地先验收 GitHub Actions Docker Compose 宝塔服务器拉镜像

1. 总体流程

  1. 开发者本地运行后端测试、前端构建、关键页面联调。
  2. 确认无问题后执行 git push 到 GitHub。
  3. GitHub Actions 自动运行测试与构建。
  4. 测试通过后构建 seedcrm-backend 和 seedcrm-frontend 镜像。
  5. 镜像推送到镜像仓库，例如 GitHub Container Registry、阿里云镜像仓库或腾讯云 TCR。
  6. GitHub Actions 通过 SSH 登录宝塔服务器，执行部署脚本。
  7. 宝塔服务器执行 docker compose pull 和 docker compose up -d。
  8. 部署脚本做健康检查，失败则保留日志并提示人工回滚。

2. 推荐目录结构

服务器上建议只保留部署文件、环境变量和数据卷，不在服务器上编译源码。

/www/wwwroot/seedcrm-deploy/
  docker-compose.yml
  .env
  deploy.sh
  rollback.sh
  logs/
  backups/

源码仍在 GitHub；宝塔服务器只拉镜像，减少服务器压力，也方便快速回滚。

3. 本地发布前检查

每次准备推送前，在本地先跑一遍：

cd E:\studio\test\seedcrm\crm_server
mvn -q test

cd E:\studio\test\seedcrm\crm_web
npm install
npm run build

本地联调建议至少检查：

  • 登录是否正常。
  • 菜单和角色权限是否正常。
  • 方案 B 分销入站 dry-run 是否不写核心业务表。
  • 上线工具摘要是否显示前端 8003、后端 8004。
  • 清理测试数据真实执行是否默认被 SEEDCRM_ALLOW_CLEAR_TEST_DATA=false 阻断。

4. 镜像仓库选择

方案优点注意事项
GitHub Container Registry和 GitHub Actions 集成最简单。国内服务器拉取可能较慢，需要测试宝塔服务器访问速度。
阿里云镜像仓库国内服务器拉取速度较好。需要配置仓库账号、命名空间和登录密码。
腾讯云 TCR国内拉取速度较好，适合腾讯云服务器。需要配置访问凭证和网络权限。

当前轻量化优先建议：如果宝塔服务器能稳定访问 GitHub 镜像仓库，就先用 GitHub Container Registry；如果拉取慢，再切到阿里云镜像仓库。

5. 宝塔服务器准备

  1. 宝塔面板安装 Docker 和 Docker Compose。
  2. 服务器放行前端端口 8003 和后端端口 8004，MySQL 不对公网放行。
  3. 创建部署目录：/www/wwwroot/seedcrm-deploy。
  4. 把部署专用 docker-compose.yml、.env、deploy.sh 放到该目录。
  5. 登录镜像仓库，例如 docker login ghcr.io 或 docker login registry.cn-hangzhou.aliyuncs.com。

6. 服务器 .env 示例

正式环境的 .env 不要提交到 GitHub，只放在服务器部署目录。

SEEDCRM_MYSQL_DATABASE=seedcrm
SEEDCRM_MYSQL_USERNAME=seedcrm
SEEDCRM_MYSQL_PASSWORD=请替换为强随机密码
SEEDCRM_MYSQL_ROOT_PASSWORD=请替换为另一个强随机密码
SEEDCRM_MYSQL_PORT=3306

SEEDCRM_SYSTEM_BASE_URL=https://crm.example.com
SEEDCRM_API_BASE_URL=https://api.example.com
SEEDCRM_SERVER_PORT=8004

SEEDCRM_FRONTEND_PORT=8003
SEEDCRM_BACKEND_PORT=8004

SEEDCRM_OPENAPI_ENABLED=false
SEEDCRM_OPENAPI_ALLOW_LOCAL_WITHOUT_TOKEN=false
SEEDCRM_ALLOW_CLEAR_TEST_DATA=false

IMAGE_REGISTRY=ghcr.io/your-org/seedcrm
IMAGE_TAG=latest

密码建议至少 24 位，包含大小写字母、数字和特殊字符。生产环境必须保持 SEEDCRM_ALLOW_CLEAR_TEST_DATA=false。

7. 部署版 docker-compose.yml 示例

服务器部署时建议使用镜像，不在服务器上 build。

services:
  mysql:
    image: mysql:8.4
    container_name: seedcrm-mysql
    restart: unless-stopped
    environment:
      MYSQL_DATABASE: ${SEEDCRM_MYSQL_DATABASE:-seedcrm}
      MYSQL_USER: ${SEEDCRM_MYSQL_USERNAME:-seedcrm}
      MYSQL_PASSWORD: ${SEEDCRM_MYSQL_PASSWORD}
      MYSQL_ROOT_PASSWORD: ${SEEDCRM_MYSQL_ROOT_PASSWORD}
      TZ: Asia/Shanghai
    command:
      - --character-set-server=utf8mb4
      - --collation-server=utf8mb4_general_ci
      - --default-time-zone=+08:00
    ports:
      - "127.0.0.1:${SEEDCRM_MYSQL_PORT:-3306}:3306"
    volumes:
      - seedcrm-mysql-data:/var/lib/mysql
    healthcheck:
      test: ["CMD-SHELL", "mysqladmin ping -h 127.0.0.1 -u$${MYSQL_USER} -p$${MYSQL_PASSWORD} --silent"]
      interval: 10s
      timeout: 5s
      retries: 20

  backend:
    image: ${IMAGE_REGISTRY}/seedcrm-backend:${IMAGE_TAG:-latest}
    container_name: seedcrm-backend
    restart: unless-stopped
    depends_on:
      mysql:
        condition: service_healthy
    environment:
      SPRING_PROFILES_ACTIVE: prod
      SEEDCRM_DATASOURCE_URL: jdbc:mysql://mysql:3306/${SEEDCRM_MYSQL_DATABASE:-seedcrm}?useUnicode=true&characterEncoding=UTF-8&connectionCollation=utf8mb4_general_ci&serverTimezone=Asia/Shanghai
      SEEDCRM_DATASOURCE_USERNAME: ${SEEDCRM_MYSQL_USERNAME:-seedcrm}
      SEEDCRM_DATASOURCE_PASSWORD: ${SEEDCRM_MYSQL_PASSWORD}
      SEEDCRM_SERVER_PORT: 8004
      SEEDCRM_SYSTEM_BASE_URL: ${SEEDCRM_SYSTEM_BASE_URL}
      SEEDCRM_API_BASE_URL: ${SEEDCRM_API_BASE_URL}
      SEEDCRM_OPENAPI_ENABLED: ${SEEDCRM_OPENAPI_ENABLED:-false}
      SEEDCRM_OPENAPI_ALLOW_LOCAL_WITHOUT_TOKEN: ${SEEDCRM_OPENAPI_ALLOW_LOCAL_WITHOUT_TOKEN:-false}
      SEEDCRM_ALLOW_CLEAR_TEST_DATA: ${SEEDCRM_ALLOW_CLEAR_TEST_DATA:-false}
      TZ: Asia/Shanghai
    ports:
      - "${SEEDCRM_BACKEND_PORT:-8004}:8004"

  frontend:
    image: ${IMAGE_REGISTRY}/seedcrm-frontend:${IMAGE_TAG:-latest}
    container_name: seedcrm-frontend
    restart: unless-stopped
    depends_on:
      - backend
    ports:
      - "${SEEDCRM_FRONTEND_PORT:-8003}:80"

volumes:
  seedcrm-mysql-data:

8. deploy.sh 示例

放在宝塔服务器 /www/wwwroot/seedcrm-deploy/deploy.sh。

#!/usr/bin/env bash
set -euo pipefail

cd /www/wwwroot/seedcrm-deploy

echo "[1/5] 检查 Docker Compose 配置"
docker compose config >/tmp/seedcrm-compose-check.yml

echo "[2/5] 拉取最新镜像"
docker compose pull

echo "[3/5] 启动容器"
docker compose up -d

echo "[4/5] 等待后端启动"
sleep 20

echo "[5/5] 健康检查"
curl -fsS http://127.0.0.1:${SEEDCRM_FRONTEND_PORT:-8003}/ >/dev/null
curl -fsS http://127.0.0.1:${SEEDCRM_BACKEND_PORT:-8004}/v3/api-docs >/dev/null || true
docker compose ps

echo "SeedCRM 部署完成"

如果正式环境关闭 Swagger/OpenAPI，/v3/api-docs 可能不可访问，这是安全预期；后续可以补一个专用健康检查接口。

9. GitHub Secrets 配置

在 GitHub 仓库进入 Settings / Secrets and variables / Actions，新增：

Secret 名称用途
REGISTRY_USERNAME镜像仓库登录用户名。
REGISTRY_PASSWORD镜像仓库登录密码或 Token。
BAOTA_HOST宝塔服务器公网 IP 或域名。
BAOTA_SSH_USERSSH 用户，建议使用非 root 部署用户。
BAOTA_SSH_KEY部署用户私钥。
BAOTA_SSH_PORTSSH 端口，例如 22。

10. GitHub Actions 示例

文件位置：.github/workflows/deploy.yml。下面以 GitHub Container Registry 为例。

name: Deploy SeedCRM

on:
  push:
    branches: [ "main" ]
  workflow_dispatch:

env:
  REGISTRY: ghcr.io
  IMAGE_NAMESPACE: your-org/seedcrm
  IMAGE_TAG: latest

jobs:
  test-build-deploy:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup JDK
        uses: actions/setup-java@v4
        with:
          distribution: temurin
          java-version: "17"
          cache: maven

      - name: Backend tests
        working-directory: crm_server
        run: mvn -q test

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: npm
          cache-dependency-path: crm_web/package-lock.json

      - name: Frontend build
        working-directory: crm_web
        run: |
          npm ci
          npm run build

      - name: Login registry
        run: echo "${{ secrets.REGISTRY_PASSWORD }}" | docker login ${{ env.REGISTRY }} -u "${{ secrets.REGISTRY_USERNAME }}" --password-stdin

      - name: Build backend image
        run: docker build -t ${{ env.REGISTRY }}/${{ env.IMAGE_NAMESPACE }}/seedcrm-backend:${{ env.IMAGE_TAG }} ./crm_server

      - name: Build frontend image
        run: docker build -t ${{ env.REGISTRY }}/${{ env.IMAGE_NAMESPACE }}/seedcrm-frontend:${{ env.IMAGE_TAG }} ./crm_web

      - name: Push images
        run: |
          docker push ${{ env.REGISTRY }}/${{ env.IMAGE_NAMESPACE }}/seedcrm-backend:${{ env.IMAGE_TAG }}
          docker push ${{ env.REGISTRY }}/${{ env.IMAGE_NAMESPACE }}/seedcrm-frontend:${{ env.IMAGE_TAG }}

      - name: Deploy to Baota server
        uses: appleboy/ssh-action@v1.2.0
        with:
          host: ${{ secrets.BAOTA_HOST }}
          username: ${{ secrets.BAOTA_SSH_USER }}
          key: ${{ secrets.BAOTA_SSH_KEY }}
          port: ${{ secrets.BAOTA_SSH_PORT }}
          script: |
            cd /www/wwwroot/seedcrm-deploy
            export IMAGE_TAG=${{ env.IMAGE_TAG }}
            bash deploy.sh

11. 正式环境发布门禁

  • GitHub Actions 后端测试必须通过。
  • 前端构建必须通过。
  • 宝塔服务器 docker compose config 必须通过。
  • 生产环境 SPRING_PROFILES_ACTIVE=prod。
  • 生产环境 SEEDCRM_OPENAPI_ENABLED=false，需要联调时短期开启并加权限控制。
  • 生产环境 SEEDCRM_ALLOW_CLEAR_TEST_DATA=false。
  • 外部回调必须使用 API 域名，例如 https://api.example.com，不能使用前端页面域名。
  • 上线前先备份数据库。

12. 回滚方式

轻量化第一版建议保留最近 3 到 5 个镜像 tag。回滚时改服务器 .env 的 IMAGE_TAG，然后执行：

cd /www/wwwroot/seedcrm-deploy
docker compose pull
docker compose up -d
docker compose logs -f backend

更稳妥的版本策略是使用 Git tag，例如 v1.0.3，正式环境不要只依赖 latest。

13. 常见风险

  • 不要把服务器 .env 提交到 GitHub。
  • 不要开放 MySQL 3306 到公网。
  • 不要在正式环境打开 SEEDCRM_ALLOW_CLEAR_TEST_DATA。
  • 不要让 GitHub Actions 使用 root 密码登录数据库。
  • 不要让外部平台回调前端域名，回调必须走 API 域名。
  • 不要只推送 latest 却没有可回滚版本。

14. 当前推荐落地顺序

  1. 先在宝塔服务器手工部署一次，确认 Compose、端口、域名和数据库都正常。
  2. 再接入 GitHub Actions，只做测试和镜像构建，不自动部署。
  3. 镜像能正常拉取后，再打开 SSH 自动部署。
  4. 最后把正式发布从 main 自动部署升级为 tag 发布，例如 v1.0.3。
