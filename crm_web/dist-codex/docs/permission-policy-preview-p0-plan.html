SeedCRM P0-2 权限策略预览详细落地方案

权限策略预览不是新的业务权限体系，而是发布前的“最终权限试算器”。它把菜单、角色、权限点、接口能力、数据范围、字段脱敏和用户上下文合并计算，回答一个最关键的问题：某个用户或角色上线后到底能看到什么、能点什么、能查到哪些数据。

P0-2 Permission RBAC + ABAC 配置发布前预览 不新增核心业务模型

1. 目标与边界

  • 目标是解决菜单管理、角色管理、权限管理修改前不可预判的问题，避免“配置以后才发现登录不了、看不到菜单、能看到但点不了、接口越权”。
  • 预览页只读，不直接修改 Clue、Customer、Order、PlanOrder、Scheduler、Salary、WeCom 的业务数据。
  • 权限预览必须复用正式 RBAC + ABAC 判定口径，不能只做前端菜单隐藏模拟。
  • 按角色预览只能展示角色默认能力；涉及 self、team、store、partner 的实际数据集合时，必须选择用户或上下文，不能默认扩大到 all。
  • 权限配置不能绕过主链路：标准链路仍为 Clue -> Customer -> Order -> PlanOrder；分销 B 链路仍为 ExternalDistributionPaidOrder -> Customer -> Order(paid) -> PlanOrder。
  • 分销已支付订单不得通过权限配置进入 Clue、公海、分配、回收或线索转化率统计。
  • 预览结果不能替代运行时鉴权；每个后端接口仍必须做 RBAC 校验和 ABAC 数据过滤。

2. 当前系统现状

结构约束

  • permission_policy 是唯一运行时生效策略表，草稿不得直接写入该表。
  • 草稿使用 JSON 快照承载菜单、角色、权限策略差异，降低 P0 阶段对现有登录链路的侵入。
  • 发布记录必须保存前后快照和 diff，支持回滚与审计。
  • 审计日志不得保存完整手机号、raw_data、密钥、签名原文等敏感内容。
  • 字段脱敏结果建议使用 VISIBLE、MASKED、HIDDEN 三态表达。

4. 接口定义

接口用途权限
GET /permission/preview/baseline读取当前生效权限快照：角色、菜单树、接口权限、策略、数据范围。permission:view
POST /permission/preview/drafts创建权限预览草稿，输入 menuDiff、roleDiff、policyDiff。permission:update
GET /permission/preview/drafts/{draftNo}查看草稿详情、差异和校验结果。permission:view
POST /permission/preview/calculate计算单个用户、角色或资源场景的最终权限，不落库。permission:check
POST /permission/preview/batch-calculate批量计算角色 × 菜单 × 接口 × 数据范围矩阵。permission:check
POST /permission/preview/validate发布前校验草稿，输出阻断项、警告项、影响范围。permission:update
POST /permission/preview/publish发布已校验草稿，事务内写入运行时权限表。permission:update
POST /permission/preview/rollback按 publishNo 回滚到上一快照。permission:update
GET /permission/preview/audit-logs查询预览、校验、发布、回滚审计记录。permission:view

5. 权限规则

最终权限计算

最终允许 = 菜单可见 AND 接口可调用 AND 数据范围命中 AND 字段策略允许
  • 菜单来自 sys_menu + sys_role_menu，菜单可见只代表入口可见，不代表接口一定可调用。
  • 接口权限来自 sys_permission + sys_role_permission 与 permission_policy 的双重约束。
  • 父级菜单只作为导航容器，由子菜单聚合生成，不单独授予业务能力。
  • 后台管理类能力必须走 PERMISSION:VIEW/UPDATE/CHECK 或同等权限点，不能开放给普通业务角色。
  • partner_app 是外部应用身份，不进入后台菜单树，只能命中 partner 数据范围。

ABAC 数据范围

范围命中规则预览要求
SELFcurrentUserId == resourceOwnerId 或绑定客户命中。必须展示绑定依据，缺少用户上下文时显示“不确定”。
TEAMresourceOwnerId in teamMemberIds。必须展示团队成员范围或提示团队上下文缺失。
STOREcurrentStoreId == resourceStoreId。必须展示门店 ID 和门店名称。
ALL仅显式授权角色可用。非 admin 获得 ALL 必须标为高风险。
PARTNERcurrentPartnerCode == resourcePartnerCode。必须展示 partner_code，禁止显示后台菜单。

高风险变更

  • 给非 admin 开全量数据、导出、raw_data 完整查看、接口配置、回放、对账、重试。
  • 删除或停用 admin 的系统管理、权限管理、回滚能力。
  • 把 integration_operator 扩大为配置渠道、签名、字段映射或直接写核心业务数据。
  • 把 partner_app 扩大为非 PARTNER 数据范围或后台菜单能力。
  • 让 PlanOrder 出现独立创建、独立履约、绕过 Order 的入口。
  • 让 distribution paid order 出现在 Clue、公海、分配、回收、线索转化率场景。

6. 流程

  1. 读取当前生效快照：从 sys_role、sys_menu、sys_permission、授权关联表和 permission_policy 生成 baseline。
  2. 创建草稿：保存菜单、角色、权限策略 diff，不写运行时表。
  3. 权限模拟：使用草稿快照构建内存策略，逐项计算菜单可见、接口权限、ABAC 数据范围、字段脱敏。
  4. 批量矩阵：覆盖角色集合、模块集合、关键路由、关键动作、典型资源上下文，输出允许、拒绝和原因。
  5. 冲突诊断：识别菜单有入口但接口无权限、接口有权限但菜单不可见、数据范围为空、敏感字段未脱敏、父子菜单不一致。
  6. 发布前校验：检查是否会锁死管理员、是否导致登录后无菜单、是否暴露核心业务接口、是否破坏主链路。
  7. 发布：事务内更新菜单、角色授权、接口授权、permission_policy，写发布记录和审计。
  8. 缓存失效：发布成功后失效权限策略缓存、菜单树缓存、用户授权快照缓存。
  9. 回滚：按发布记录的 before 快照恢复，仍需走校验和审计。

7. 前端体验方案

角色视角

选择角色后，同屏展示可见菜单、按钮、接口、数据范围、字段脱敏和高风险能力摘要。进入编辑模式后原地打开右侧抽屉，不跳到页面顶部。

菜单视角

新增菜单时通过树选择父级菜单，自动带出模块、路由和权限码候选；不允许手填顶级菜单名称，避免录入不确定性。

变更预览

保存前展示配置前后差异：影响哪些角色、菜单、接口、字段、数据范围。高风险变更必须填写原因和确认词。

冲突面板

红色为阻断项，橙色为高风险确认项，蓝色为信息提示。每条问题必须说明原因和修复建议，不能只弹 toast。

8. 发布前强制校验

  • 至少一个 admin 保留 PERMISSION:VIEW/UPDATE/CHECK、系统管理菜单和回滚能力。
  • 登录、刷新登录态、获取当前用户信息接口不能被预览草稿阻断。
  • Clue、Customer、Order、PlanOrder 主链路关键动作不能形成无授权孤岛。
  • PlanOrder 创建必须绑定已有 Order 上下文，不能独立存在。
  • Scheduler 只能做队列、重试、补偿和对账，不得通过权限配置绕过入站服务直接写 Customer、Order、PlanOrder。
  • integration_operator 不得获得配置修改和核心业务写入权限。
  • partner_app 不得获得后台菜单和非 partner 数据范围。
  • raw_data、手机号、金额、外部会员 ID 等敏感字段必须有脱敏策略。

9. 测试与验收

测试项验收标准
按用户预览能选择真实用户，生成最终菜单树、按钮、接口、数据范围和字段脱敏结果。
按角色预览能选择角色生成默认能力，并提示缺少用户上下文时无法判断实际 self/team/store/partner 数据集合。
ABAC 正反例self、team、store、all、partner 每类范围至少有一个允许和一个拒绝用例。
菜单接口一致性菜单可见但接口不可调用、接口可调用但菜单不可见都能被诊断出来。
分销 B 约束distribution paid order 不进入 Clue，不参与公海、分配、回收、线索转化率。
PlanOrder 约束预览里不能出现独立创建 PlanOrder 的能力，必须绑定 Order 和服务流程。
集成角色约束integration_admin 与 integration_operator 能力边界清晰，operator 不能改配置或写核心业务数据。
partner_app 隔离只能访问自身 partner_code 范围数据，无后台菜单。
敏感字段手机号、raw_data、金额、外部会员 ID 能预览可见、脱敏、隐藏效果。
发布事务发布失败时运行时权限不变；发布成功后缓存即时失效。
审计预览、校验、发布、回滚都有审计记录，敏感字段脱敏。
体验编辑角色或菜单不跳顶部，新增菜单父级必须树选择，冲突可停留查看并逐条处理。

10. 落地顺序

  1. 先修复权限中心现有中文乱码和编辑体验问题，避免在坏体验上叠新能力。
  2. 实现 baseline 快照接口增强和前端只读预览页面。
  3. 实现单场景 calculate，不落库，复用现有 PermissionService.check 口径。
  4. 实现批量矩阵和冲突诊断，覆盖菜单、接口、数据范围、字段脱敏。
  5. 实现草稿、校验、发布、审计和缓存失效。
  6. 实现回滚能力和发布记录查询。
  7. 接入菜单管理、角色管理、权限管理的保存前预览入口。
