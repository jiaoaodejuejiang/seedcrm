SeedCRM P0-1 统一字典中心详细落地方案

统一字典中心用于“编码值的中文展示、筛选选项、状态色、排序、启停、字段脱敏分组、来源视角分组”。它不是核心业务状态机，也不能替代权限、流程、幂等、资金边界和接口安全。

P0-1 系统能力 配置优先 不新增核心模型 低频高影响

1. 字典中心边界

  • 字典可以控制显示名称、状态颜色、筛选选项、排序、启停、说明文案。
  • 字典不能改变核心状态流转，例如 Order 仍只能由受控服务从 paid / used / cancelled / refund_pending / refunded 中流转。
  • 字典不能新增核心模型，也不能允许外部系统直接创建 Customer / Order / PlanOrder。
  • 字典不能替代 RBAC / ABAC；字典项可见性必须继续受角色、数据范围、字段脱敏控制。
  • 字典不能保存密钥、token、secret、支付证书、企业微信 secret、抖音 auth_code；这些属于 Provider / 系统配置敏感项。
  • 字典启停不得删除历史数据，只影响页面新筛选、新录入和展示。
  • 组织、人员、门店、商品等有层级和权限的数据不作为普通字典，必须使用独立主数据或配置模块。
  • 高风险字典不做即时随手生效，必须经过变更预览、原因填写、确认词和审计。

3. 数据结构

表字段说明
system_dictionaryid、dict_code、dict_name、module_code、risk_level、value_mode、scope_mode、enabled、editable、builtin、description、version_no、created_at、updated_at字典分组。builtin 表示系统内置，不允许删除；editable 控制是否允许后台编辑。
system_dictionary_itemid、dict_code、item_code、item_value、display_name、short_name、parent_code、style_token、sort_no、scope_type、scope_id、enabled、builtin、sensitive_level、metadata_json、description、created_at、updated_at字典项。item_code 是稳定业务编码，不随中文变化；scope 支持后续门店、伙伴、租户扩展。
system_dictionary_usageid、dict_code、item_code、usage_module、usage_scene、field_name、reference_count、block_disable、description字典使用关系，用于修改前提示影响范围。
system_dictionary_draftid、batch_no、dict_code、draft_type、payload_json、status、operator_user_id、reason、created_at待发布变更草稿。P0 可先用于高风险变更和导入预览。
system_dictionary_change_logid、dict_code、item_code、change_type、before_json、after_json、operator_role、operator_user_id、reason、created_at变更审计，支持追踪和后续回滚。

唯一约束建议：system_dictionary.dict_code 唯一，system_dictionary_item(dict_code, item_code, scope_type, scope_id) 唯一。业务表保存稳定编码 item_code，不保存展示文案。

4. 接口定义

/dictionary/batchGET按 typeCodes 批量查询运行时字典。按角色和作用域过滤。

/dictionary/import/commitPOST确认导入。需要导入预览批次号和确认词。

5. 权限规则

  • dictionary:view：admin、integration_admin、integration_operator、clue_manager、finance 可查看必要字典。
  • dictionary:update：默认仅 admin；integration_admin 可维护接口、Provider、外部状态类字典。
  • dictionary:import / dictionary:export：默认仅 admin。
  • 高风险字典包括：order.status、order.type、plan_order.status、customer.source、provider.execution_mode、field.mask_level。
  • 高风险字典修改必须展示影响范围、要求填写修改原因、记录审计日志。
  • integration_operator 不允许修改字典，只能查看用于排查接口日志和异常队列。
  • partner_app 不能访问后台字典管理接口，只能通过 Open API 获得允许暴露的枚举说明。
  • 字典项 metadata_json 中如果包含敏感字段，必须按字段脱敏策略返回。
  • 缓存 key 必须包含 typeCode、scopeType、scopeId、roleCode、dataScope；避免 ABAC 作用域串数据。

6. 流程

新增字典项流程

  1. 管理员进入系统设置 / 字典管理。
  2. 选择字典分组，点击新增字典项。
  3. 填写 item_value、display_name、状态色、排序、说明。
  4. 系统校验编码唯一、是否命中保留字、是否影响核心枚举边界。
  5. 低风险字典保存后写入审计日志并刷新缓存；高风险字典先进入待发布区。

修改高风险字典流程

  1. 点击编辑后，系统展示引用页面、接口、字段和历史数据量。
  2. 只能修改 display_name、style_token、sort_no、description；内置 item_code 不允许修改。
  3. 提交前展示差异摘要和风险说明。
  4. 输入确认词 UPDATE_DICTIONARY 或具体字典编码，并填写修改原因。
  5. 保存为草稿后进入待发布区；发布时写审计、刷新缓存；如失败必须回滚事务。

运行时解析流程

  1. 后端列表接口继续返回稳定编码值。
  2. 前端通过批量字典解析或本地缓存把编码转为中文。
  3. 关键导出和对外接口由后端统一解析，避免前端导出不一致。
  4. 如果字典项不存在，页面显示“未知编码：xxx”，并提示管理员补充字典。

7. 页面交互

  • 左侧为字典分组树，右侧为字典项表格。
  • 顶部显示风险等级、引用数量、最近修改人、最近修改时间。
  • 字典项表格列：编码、显示名称、短名称、状态色、排序、启停、内置、说明、操作。
  • 编辑使用抽屉，危险修改使用弹窗确认。
  • 停用字典项不删除历史，只从新录入和筛选中隐藏。
  • 提供“查看引用”按钮，显示哪些页面、接口、字段正在使用。
  • 提供“导入预览”，先展示新增、修改、冲突，不允许直接覆盖。
  • 提供“未知编码看板”，列出最近接口和页面中出现但未配置的编码。
  • 默认只读浏览，点击“进入编辑模式”后才允许新增、编辑、排序、启停。
  • 高风险变更进入底部“待发布变更区”，统一预览影响后发布。

8. 兼容迁移

  • 第一阶段不强制改业务表，只新增字典表和默认种子数据。
  • 前端已有硬编码中文映射逐步迁移到字典解析，迁移期间保留本地 fallback。
  • 后端导出、审计、异常队列优先使用字典解析，减少编码裸露。
  • 历史状态值不改库，只通过字典展示中文。
  • 若发现代码中存在文档外状态值，进入“未知编码看板”，不自动当作合法业务状态。
  • 首期缓存可用本地 Caffeine / Spring Cache；多实例上线后再通过 Redis Pub/Sub 或消息表做缓存失效广播。

9. 验收门禁

  • 客资、会员、订单、PlanOrder、调度、异常队列至少 80% 状态字段不再裸露编码。
  • 新增来源展示不需要改前端硬编码。
  • Order / PlanOrder 核心状态 item_value 不可通过字典管理修改。
  • 停用字典项不会删除历史数据，也不会破坏列表展示。
  • 高风险字典修改必须有引用范围、差异摘要、确认词、修改原因和审计记录。
  • 字典解析失败时显示未知编码并可追踪，不静默显示空值。
  • 所有字典接口必须经过 RBAC，敏感 metadata 必须脱敏。
  • 字典运行时缓存更新后能按 typeCode 精准失效，不全量清空。
