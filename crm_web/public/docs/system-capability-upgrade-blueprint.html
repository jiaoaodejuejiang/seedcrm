SeedCRM 系统能力升级蓝图与 P0 落地计划

目标：后续业务变化优先通过系统配置、权限策略、接口适配、流程规则、表单模板完成；确需代码增强时，优先增强系统能力，而不是把单次业务需求写死。

约束优先 主链路不变 配置化优先 工作人员体验优先

1. 不可突破的业务边界

  • 标准链路固定为 Clue → Customer → Order → PlanOrder。
  • 分销方案 B 固定为 ExternalDistributionPaidOrder → Customer → Order(paid) → PlanOrder。
  • distribution 已支付订单不进入 Clue，不参与客资公海、自动分配、线索转化率统计。
  • 外部系统不得直接创建 Customer / Order / PlanOrder，必须通过受控 Open API / Scheduler 复用入站服务。
  • Scheduler 不得绕过入站服务直接写 Customer / Order / PlanOrder。
  • SeedCRM 不承接外部分销佣金、提现、退款资金流，只承接已支付数据、预约排档、门店履约、状态回推。
  • 权限必须覆盖 RBAC、ABAC、字段脱敏、partner 数据范围；raw_data 默认脱敏。

2. 成熟系统参考，但不重复造轮子

参考对象可借鉴能力SeedCRM 的取舍
RuoYi菜单、角色、部门、字典、参数、日志、数据权限。借鉴后台基础设施，不直接照搬代码生成式业务开发。
芋道 / ruoyi-vue-pro权限、工作流、动态表单、系统配置、审计、低代码思想。学习配置治理和流程治理，但主链路约束必须由代码和数据库兜底。

参考资料：RuoYi 文档、芋道功能列表、Flowable BPMN Introduction、Element Plus Table、Element Plus Form。

3. 系统能力分层

层级能力目标
基础治理层字典、参数、配置版本、审计、导入导出、环境保护。把状态、来源、字段显示、危险开关统一管理。
权限策略层菜单、按钮、接口、数据范围、字段脱敏、partner 范围。让权限配置可视、可测、可回归。
接口适配层Provider、签名、字段映射、状态映射、dry-run、回调日志、异常队列。新增渠道优先配置接入，不写散落业务逻辑。
调度与队列层任务模板、执行模式、频率、重试、Outbox、对账、状态回查。调度只做同步、补偿、重放和回推，不越权写核心表。
流程与规则层轻量状态机、节点动作、表单模板、规则试算、命中日志。流程可配置，但真实状态变更仍调用受控服务。
工作人员体验层今日待办、统一列表、统一表单、状态下一步、配置防误触。让员工少理解系统结构，多按任务完成工作。

5. P0 接口定义

接口方法目标约束
/system/dictionariesGET / POST / PUT字典分组和字典项维护。必须有审计；生产关键字典修改需确认。
/system/configs/versionsGET查看配置版本和差异。敏感值脱敏。
/system/configs/rollbackPOST回滚配置到指定版本。高风险配置需要二次确认和权限。
/permission/policies/previewPOST预览角色能看哪些菜单、按钮、数据和字段。不落库，用于配置前验证。
/integration/providers/validatePOSTProvider 配置预检。默认 dry-run，不调用真实写入接口。
/integration/mappings/dry-runPOST字段映射、状态映射、幂等匹配预览。不得写 Customer / Order / PlanOrder。
/scheduler/jobs/dry-runPOST调度任务预检。真实执行前必须先 dry-run。
/workflow/definitions/simulatePOST流程路径和节点权限模拟。只模拟，不改变业务状态。
/rules/simulatePOST规则试算，返回命中条件和结果。不落库，保留试算日志。
/workbench/todosGET今日待办聚合。按 RBAC/ABAC 返回可处理事项。

6. P0 权限规则

  • admin：可维护系统字典、配置版本、权限策略、Provider、调度、流程定义，但生产危险操作仍需要确认和审计。
  • integration_admin：可配置渠道、签名、字段映射、状态映射、调度策略，不可直接修改核心业务数据。
  • integration_operator：可查看日志、异常、重试结果，可发起受控重试，不可修改 Provider 密钥和核心数据。
  • clue_manager：可配置客资分配规则、值班客服、客资范围，不可处理 distribution paid order 进入 Clue。
  • store_service：只能处理门店范围内 Order / PlanOrder / 服务单，不可查看外部 raw_data 完整内容。
  • finance：可查看 finished 后内部门店薪酬和结算，不承接外部分销佣金、提现、退款资金流。
  • private_domain_service：可查看绑定客户、会员视角、企微关系和标签，不可直接创建 Customer / Order。
  • partner_app：非后台角色，仅能访问自身 partner_code 范围 Open API。

每个接口都必须经过 RBAC；每个列表、详情、导出、raw_data 查看都必须经过 ABAC 和字段脱敏。

7. P0 流程

新增业务需求处理流程

  1. 先判断是否影响核心链路：Clue、Customer、Order、PlanOrder。
  2. 如果不影响核心链路，优先映射到字典、配置、权限、表单、流程、规则、Provider。
  3. 如果系统能力不足，先增强系统能力，再承接业务需求。
  4. 所有接口联调先 dry-run，再 MOCK，再 LIVE。
  5. 所有高风险操作必须有环境条、影响预览、二次确认、审计日志、回滚方案。

外部渠道接入流程

  1. 创建 Provider 配置，配置签名、限流、回调路径、执行模式。
  2. 配置字段映射、状态映射、订单类型映射。
  3. 用 dry-run 预览 Customer 匹配、Order 幂等、字段落点。
  4. 通过 MOCK 模式完成异常队列、重试、对账、回推闭环。
  5. LIVE 模式只能通过统一入站服务进入主链路。

工作人员操作流程

  1. 进入今日待办，不要求员工理解全部菜单结构。
  2. 按待办进入客资、订单、排档、服务单、异常队列。
  3. 页面展示当前状态、阻塞原因、下一步主操作。
  4. 危险操作进入抽屉或弹窗，显示影响范围和确认词。
  5. 操作完成后提示下一步，而不是只显示“成功”。

8. 统一体验规范

场景规范
列表页状态快捷筛选 + 高级筛选 + 列设置 + 分页 + 固定操作列 + 最后刷新时间。
表格操作一行一个主操作，次操作收进“更多”，危险操作固定在底部并用红色。
编辑低风险字段可行内编辑；会改变业务阶段、归属、流程的操作必须抽屉确认。
配置页顶部显示当前环境、运行模式、最近修改人、最近修改时间；保存前显示差异摘要。
调度页预检和真实执行视觉分离，真实执行必须 warning / danger，且需要影响范围。
空状态不要只显示暂无数据，要给下一步入口，例如“去联调工作台验证接口”。
错误提示包含问题是什么、数据是否已写入、下一步去哪处理。

9. 落地顺序

  1. P0-1：统一字典中心。先覆盖来源、状态、会员角色、异常类型、字段脱敏、tab 分组。
  2. P0-2：权限策略预览。角色选择后可看到菜单、按钮、接口、数据范围、字段范围。
  3. P0-3：配置版本和回滚。所有高风险配置保存前有差异，保存后可追踪，可回滚。
  4. P0-4：Provider 接入标准化。抖音、分销、企微共用 Provider + Mapping + DryRun 模式。
  5. P0-5：调度安全增强。任务模板、dry-run、真实执行确认、日志、重试、Outbox 统一。
  6. P0-6：今日待办工作台。先聚合高频任务，不大改原业务页面。
  7. P1：轻量流程和规则试算。服务单、异常处理、薪酬、分配、标签逐步规则化。
  8. P1：通用动态表单。从服务单模板扩展，不先做全量低代码。

10. 验收门禁

  • 新增来源不改代码即可完成展示、筛选、状态中文化和字段脱敏。
  • 新增外部渠道能通过 Provider 配置 + Mapping dry-run 完成联调预览。
  • 任一真实调度执行前必须有 dry-run 记录和影响范围。
  • 权限策略能预览角色可见菜单、按钮、接口、数据范围、字段范围。
  • 所有配置变更都有版本、差异、操作者、原因和回滚能力。
  • 分销 paid order 永远不进入 Clue，且 Scheduler 不直接写核心表。
  • 工作人员能从今日待办完成主要任务，不必须记住复杂菜单路径。
