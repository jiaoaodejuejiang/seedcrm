SeedCRM P0-3 配置版本和回滚详细落地方案

配置版本和回滚是系统配置的发布安全层。它不替代业务模块，也不改变核心链路，而是让域名、Provider、回调、调度、权限、字典高风险项、支付和企微配置都具备草稿、差异、dry-run、发布、回滚、审计和敏感值保护。

P0-3 SystemConfig 配置发布中心 高风险先预览 不新增核心业务模型

1. 目标与边界

  • 目标是解决高风险配置“直接保存生效、难以回滚、差异不可见、敏感值容易泄露、上线联调不确定”的问题。
  • 配置版本中心只回滚配置，不回滚已经发生的 Clue、Customer、Order、PlanOrder、薪酬、服务单、回调日志和异常队列业务数据。
  • 标准主链路仍为 Clue -> Customer -> Order -> PlanOrder；分销 B 链路仍为 ExternalDistributionPaidOrder -> Customer -> Order(paid) -> PlanOrder。
  • Scheduler、Open API、Provider 配置不得允许任何链路绕过统一入站服务直接写 Customer、Order、PlanOrder。
  • distribution paid order 在任何配置版本下都不能进入 Clue、公海、分配、回收或线索转化率统计。
  • 外部分销佣金、提现、退款资金流不进入 SeedCRM；配置中心只管理必要的接口、回调、状态展示和履约回推配置。
  • 密钥、token、auth_code、raw_data、手机号、支付流水等敏感信息在列表、差异、审计、日志中默认脱敏。

2. 当前系统现状

能力现状P0-3 补齐方向
基础配置已有 system_config 和 system_config_change_log。增强为可发布版本：草稿、差异、回滚、发布记录、敏感值脱敏。
域名配置已有系统基础域名和 API 域名，并可派生分销入站、Swagger/OpenAPI 地址。域名改动必须版本化；其他页面只展示拼接结果，不到处编辑基础域名。

回调配置已有 integration_callback_config，包含回调地址、签名、token、aes_key。回调路径和签名配置必须版本化；敏感字段只展示指纹或是否已设置。
支付配置已有微信支付/代付测试服务，能校验必填和回调拼接。支付相关回调、商户号、证书、密钥引用纳入版本治理，但不进入外部分销资金流。
企微配置已有 wecom_app_config，包含 app_secret、auth_code、suite_ticket、回调 token 等。企微配置必须支持敏感值脱敏、auth_code 时效、回调地址由 API 域名拼接。
上线工具已有上线检查和初始化/清理测试数据能力。配置发布中心应与上线检查联动，PROD 禁止清理测试数据，LIVE 发布必须先预检。

敏感值策略

  • 密钥、token、auth_code、证书私钥、签名 key 必须加密存储；应用主密钥来自环境变量或密钥管理服务，不进入数据库。
  • 差异预览只展示“未设置、已设置、已变更、尾号/指纹”，不展示完整明文。
  • auth_code 必须带状态、接收时间、过期时间或 TTL 计算，过期或重复使用时阻断 LIVE 发布。
  • dry-run 结果只保存摘要，不写 Customer、Order、PlanOrder，也不触发真实外部写接口。

4. 接口定义

接口用途权限
GET /system-config/versioning/current读取当前环境已发布配置版本和环境条信息。setting:view
GET /system-config/versioning/versions查询版本列表、状态、发布人、影响范围。setting:view
POST /system-config/versioning/drafts创建配置草稿，包含 systemConfigDiff、providerDiff、callbackDiff。setting:update 或 integration:config
GET /system-config/versioning/drafts/{versionNo}查看草稿详情、差异和校验结果，敏感值脱敏。setting:view
POST /system-config/versioning/dry-run预演配置影响，不落库、不触发真实外部写入。setting:debug 或 integration:debug
POST /system-config/versioning/validate发布前校验域名、Provider、签名、auth_code 时效、调度影响。setting:update
POST /system-config/versioning/publish发布 VALIDATED 草稿，事务内更新运行时配置表并写发布记录。setting:update
POST /system-config/versioning/rollback按 publishNo 或 versionNo 生成回滚草稿并发布新版本。setting:update
POST /system-config/versioning/environment/switch切换 LOCAL/DEV/TEST/STAGING/PROD 环境配置。setting:update
GET /system-config/versioning/audit-logs查询配置治理审计。setting:view

5. 权限规则

角色允许能力限制
admin查看、创建草稿、dry-run、校验、发布、回滚、环境切换。PROD 高风险操作仍需确认词和审计。
integration_admin维护 Provider、回调、Scheduler 集成配置，可 dry-run 和校验。发布到 PROD 建议仍需 admin 审批或复核。
integration_operator查看配置摘要、调试结果、dry-run 结果、异常和重试结果。不可保存密钥、不可发布、不可回滚。
partner_app无后台配置版本权限。只能访问自身 partner_code 范围 Open API。
门店、客服、财务仅可读取运行时必要开关，且必须脱敏。不可进入配置发布中心。

ABAC 规则

  • scope_type、scope_id 必须参与配置读取和发布，禁止用前端请求参数提升到 GLOBAL。
  • Provider 配置按 provider_code、partner_code 隔离；PARTNER 范围只能看自身。
  • PROD 发布、回滚、环境切换、清理测试数据必须有确认词和审计；PROD 禁止清理测试数据。
  • 配置版本接口不得影响登录、鉴权、菜单加载；登录所需基础配置必须保留安全默认值。

6. 流程

  1. 创建草稿：读取当前运行时配置快照，保存 diff，敏感字段立即加密和脱敏。
  2. 保存草稿：草稿不生效，页面明确显示“未发布变更”。
  3. 差异预览：展示新增、修改、停用、删除、风险等级、影响页面、影响接口、影响调度任务、最终拼接 URL。
  4. dry-run：只做格式校验、签名模拟、auth_code 时效检查、调度影响分析、主链路写入风险检查，不落业务库。
  5. 发布前校验：检查环境、域名、URL、密钥存在性、auth_code、Provider、Scheduler、权限和字典引用。
  6. 发布：事务内写配置版本状态、更新目标配置表、写发布记录、写审计；失败整体回滚。
  7. 缓存失效：事务提交后失效 system_config、Provider、Scheduler 客户端、Open API 签名配置缓存。
  8. 回滚：选择历史版本后先生成回滚草稿，再预览差异和风险，通过校验后发布为新版本。
  9. 环境切换：LOCAL/DEV/TEST/STAGING 可按权限切换；PROD 必须确认词，并阻断测试数据清理。
  10. 上线检查：LIVE 模式发布后联动上线检查，验证域名、OpenAPI、Provider、回调、调度、权限预览是否通过。

7. 版本化范围

配置类型处理方式说明
系统基础域名、API 域名、回调基础地址必须版本化影响登录、扫码服务单、OpenAPI、回调、Swagger 展示。
抖音、分销、企微 Provider必须版本化影响外部入站、回推、授权、签名、核销、状态回查。
字段映射、状态映射、订单类型映射必须版本化影响 Customer 匹配、Order 幂等和异常队列。
Scheduler 任务参数必须版本化影响同步频率、重试、对账、回推量。
权限策略发布必须版本化必须调用 P0-2 权限预览。
字典高风险项必须版本化来源、订单状态、PlanOrder 状态、脱敏规则等。
支付回调和商户配置必须版本化只管理 SeedCRM 识别和回调，不承接外部分销资金流。
普通展示文案、颜色、排序只审计不影响业务流转，可低风险保存。
个人偏好、列表列宽、查询默认项只审计或不审计不进入配置发布中心。
数据库连接、加密主密钥、核心模型定义禁止在线改通过部署和运维流程处理。

8. 前端体验方案

环境条

页面顶部固定展示环境、运行模式、当前版本、未发布变更数、最近发布人和最近发布时间。

配置卡片

主视图按业务分组展示配置卡片，不以 key/value 表格为核心。技术 key 放到“技术详情”折叠区。

差异预览

按“变更摘要、字段对比、影响范围、风险检查”四段展示；拼接地址展示最终变化并支持复制。

回滚中心

回滚从发布记录详情进入，先生成回滚草稿，不能一键直接覆盖线上。

9. 发布前阻断项

  • PROD 环境 API 域名仍是 127.0.0.1、localhost 或内网不可访问地址。
  • LIVE Provider 缺少 appId、secret、baseUrl、签名配置、有效 auth_code 或 token。
  • auth_code 已过期、重复使用、来源回调无 traceId 或 partner_code 缺失。
  • Scheduler 真实执行未 dry-run，或预计调用量异常放大。
  • 状态映射会让 distribution paid order 进入 Clue。
  • 字段映射缺少幂等键 external_partner_code + external_order_id。
  • 权限策略会导致 admin 无法进入系统管理、权限管理或回滚中心。
  • 敏感字段将在审计、差异、日志或列表中明文展示。

10. 测试与验收

测试项验收标准
草稿隔离创建和保存草稿不改变 system_config、Provider、Callback 生效值。
差异预览能展示字段级差异、影响范围、风险等级和最终拼接 URL。
dry-run全程不落核心业务表，不触发真实外部写入。
敏感值数据库加密，列表、差异、审计和日志均脱敏。
auth_code过期、重复使用、状态异常会阻断 LIVE 发布或换 token。
发布事务发布成功后新配置生效并失效缓存；发布失败时旧配置保持不变。
回滚回滚生成新版本和审计记录，不覆盖历史版本。
权限integration_operator 只能查看和 dry-run，不可 publish/rollback。
PROD 安全PROD 发布必须确认词，PROD 禁止清理测试数据。
主链路保护配置变更不能让 Scheduler 或 Open API 绕过入站服务直接写核心表。

11. 落地顺序

  1. 修复系统配置、支付、企微、调度相关页面中的历史中文乱码和敏感字段展示问题。
  2. 实现配置版本 baseline 和当前环境条，只读展示当前已发布版本。
  3. 实现配置草稿和差异预览，先覆盖域名、Provider、回调、Scheduler 四类高风险配置。
  4. 实现 dry-run 和 validate，重点覆盖 auth_code 时效、域名、签名、幂等键、状态映射、调度影响。
  5. 实现发布事务、审计、缓存失效和失败回滚。
  6. 实现历史版本、回滚草稿和回滚发布。
  7. 接入 P0-1 字典高风险项和 P0-2 权限策略发布。
  8. 接入上线检查、Docker/宝塔部署文档和 OpenAPI/Swagger 地址展示。
