SeedCRM P0-4 Provider 接入标准化详细落地方案

Provider 接入标准化建议在产品侧命名为“渠道接入中心”。它把抖音、分销、企微、支付等外部系统统一收敛到 Provider + Mapping + Dry-run + Callback + Scheduler + Outbox + Exception Queue 的接入框架里。

P0-4 Scheduler Provider Dry-run 不落库 不新增核心业务模型

1. 目标与边界

  • 目标是把外部接入从零散页面和接口字段，升级为可配置、可联调、可监控、可审计、可回滚的渠道接入中心。
  • 标准客资链路仍为 Clue -> Customer -> Order -> PlanOrder。
  • 分销 B 链路仍为 ExternalDistributionPaidOrder -> Customer -> Order(paid) -> PlanOrder。
  • 抖音客资同步只进入 Clue，不直接创建 Customer、Order、PlanOrder。
  • 分销 paid order 不进入 Clue，只能通过统一入站服务创建或匹配 Customer，并创建 Order(paid)。
  • 企微关系只能绑定 Customer，不得通过企微回调直接创建 Customer。
  • 支付接入只用于 SeedCRM 需要识别的支付状态和回调，不承接外部分销佣金、提现、退款资金流。
  • Open API Controller 和 Scheduler 都不得绕过统一入站服务直接写 Customer、Order、PlanOrder。
  • Mapping 只能做字段、状态、类型、金额单位转换，不能配置“落库动作”或新业务链路。

结构约束

  • client_secret、token_value、aes_key、auth_code、access_token、refresh_token 必须密文存储，接口只返回 masked。
  • provider_code + idempotency_key 和 provider_code + event_id 应作为幂等唯一约束。
  • Mapping 规则不允许定义“直接写核心表”的动作，只能提供转换结果给受控服务校验。
  • Outbox 事件 V1 固定支持 crm.order.used，由 PlanOrder finished 后异步回推。

ABAC 规则

  • PARTNER 范围必须满足 currentPartnerCode == resourcePartnerCode。
  • Provider 管理通常只允许 ALL 范围后台管理角色，业务数据查看再按 self/team/store/partner 过滤。
  • 异常队列中的手机号、raw_payload、签名、token、auth_code 默认脱敏。
  • integration_operator 的异常重试必须调用受控 replay 服务，不得直写 Customer、Order、PlanOrder。

6. 流程

标准入站流程

  1. 外部请求进入 /open/distribution/events 或统一 callback。
  2. 读取 Provider 配置，判断 MOCK/LIVE、enabled、签名模式、密钥状态。
  3. LIVE 模式校验 timestamp、nonce、signature、idempotency_key、partner_code。
  4. 写入 callback event log，记录 traceId、body_hash、验签状态、幂等状态。
  5. 幂等命中则返回已有处理结果，不重复落业务。
  6. 根据 Mapping 做字段、状态、金额单位、订单类型映射。
  7. dry-run 到此结束，只返回预览，不落核心表。
  8. LIVE 处理调用统一入站业务服务。
  9. distribution.order.paid 创建或匹配 Customer，并创建 Order(paid)。
  10. cancelled/refund_pending/refunded 只能更新已有 Order 的外部状态和履约影响。
  11. 失败写入异常队列，状态为 OPEN。
  12. 成功更新 callback log 的 related_customer_id、related_order_id、process_status。

Scheduler 补偿流程

  1. Scheduler 只负责拉取、回查、对账、重试、回推。
  2. 状态回查或对账发现变化时生成标准事件。
  3. 事件必须走 DistributionEventIngestService.replayFromScheduler 可信重放。
  4. 可信重放可跳过外部请求签名头，但必须保留 partner_code、idempotency_key、raw_data、callback log。
  5. 重放成功后异常记录标记 HANDLED，失败回到 OPEN。

Outbox 回推流程

  1. PlanOrder finished 后，Order 转 used。
  2. 先写 scheduler_outbox_event，事件类型 V1 固定为 crm.order.used。
  3. Scheduler 异步读取 PENDING 事件。
  4. LIVE 模式生成 timestamp、nonce、signature 后回推外部 Provider。
  5. 成功标记 SUCCESS，失败进入重试，超过阈值进入 DEAD_LETTER。
  6. 外部回推失败不回滚本地业务主链路。

7. 前端体验方案

渠道总览

展示渠道名称、类型、MOCK/LIVE、启停、接入进度、最近入站、异常数、健康状态和最近失败原因。

接入向导

按“渠道接入、授权、安全、字段映射、状态映射、dry-run、MOCK 验证、上线检查、监控”组织。

拼接地址

API 域名只读展示，入站地址、状态回查、对账、OpenAPI 地址用卡片展示，支持复制和连通性测试。

运行监控

按 traceId、外部订单号、幂等键、状态、异常类型检索；异常详情用时间线展示接收、验签、映射、匹配、建单、回推。

8. 上线阻断项

  • LIVE 模式缺少 AppSecret、timestamp、nonce、signature 校验。
  • API 域名是 localhost、127.0.0.1、非 HTTPS 或不可访问。
  • dry-run 会写核心业务表或调用真实外部接口。
  • 状态映射会让 distribution paid order 进入 Clue。
  • 字段映射缺少幂等键 external_partner_code + external_order_id。
  • Scheduler 或人工重试绕过统一入站服务直接写 Customer、Order、PlanOrder。
  • 敏感字段将在日志、审计、异常列表或 dry-run 中明文展示。
  • 切换 LIVE 前没有最近一次通过的 dry-run 或 MOCK 验证记录。

10. 落地顺序

  1. 先整理现有三方接口、回调接口、接口调试、任务调度页面，统一归入“渠道接入中心”。
  2. 修复现有接入相关页面中文乱码、敏感字段展示、MOCK/LIVE 视觉提示不足的问题。
  3. 标准化 Provider 详情和渠道总览，展示健康状态、最近入站、异常数、最近失败原因。
  4. 新增 integration_mapping_rule，先迁移分销订单类型、状态映射、字段映射。
  5. 扩展统一 dry-run 工作台，覆盖抖音、分销、企微、支付四类 Provider。
  6. 接入 P0-3 配置版本和回滚，Provider/Mapping/Callback/Scheduler 高风险变更先草稿后发布。
  7. 完善上线检查、traceId 全链路、Outbox 和异常队列监控。
  8. 补齐自动化测试和 OpenAPI/Swagger 联调说明。
