SeedCRM P0-5 调度安全增强详细落地方案

调度安全增强建议在产品侧命名为“调度与队列中心”。它统一治理任务配置、dry-run、真实执行确认、队列重试、Outbox 回推、异常可信重放、状态回查、对账拉取和上线前检查。

P0-5 Scheduler Outbox Exception Queue Dry-run 优先 不绕过主链路

1. 目标与边界

  • 标准客资主链路仍然是 Clue -> Customer -> Order -> PlanOrder。
  • 分销 B 方案主链路仍然是 ExternalDistributionPaidOrder -> Customer -> Order(paid) -> PlanOrder。
  • Scheduler 只能做同步、队列、重试、补偿、状态回查、对账拉取和 Outbox 异步回推。
  • Scheduler 不允许直接新增、修改、删除 Customer、Order、PlanOrder。
  • Open API Controller 和 Scheduler 必须复用统一入站业务服务，禁止形成两套写核心业务表的逻辑。
  • 抖音客资增量拉取任务只允许进入 Clue，不创建 Customer、Order、PlanOrder。
  • 分销 paid order 不进入 Clue，只能通过 distribution.order.paid 事件创建或匹配 Customer 并创建 Order(paid)。
  • cancelled、refund_pending、refunded 只能更新已有订单外部状态和履约影响，不能反向创建订单。
  • Outbox 回推失败不能回滚本地已完成履约，本地事务不得同步依赖外部系统响应成功。
  • 所有真实执行、批量重试、异常重放、死信处理、切换 LIVE 必须具备权限、审计、二次确认和可追踪 traceId。

数据约束

  • scheduler_job_run.run_id 必须全局唯一。
  • scheduler_job_run(job_code, idempotency_key) 必须唯一，防止重复真实执行。
  • scheduler_outbox_event.event_key 必须唯一，防止履约完成回推重复创建。
  • distribution_exception_record 必须保留原始入站 traceId，异常重放只能从记录读取原 payload。
  • raw_payload、手机号、token、secret、auth_code、支付流水、签名字段在查询和日志中默认脱敏。
  • 多实例部署时，任务锁、防重放 nonce、幂等键不能只依赖 JVM 内存，必须使用 MySQL 条件更新、唯一索引或 Redis。

4. 接口定义

接口层要把“查看、预检、准备执行、确认执行、审计追踪”拆开，不能提供一步式高风险真实执行入口。

任务查询与 dry-run

GET  /scheduler/jobs
GET  /scheduler/jobs/{jobCode}
GET  /scheduler/jobs/{jobCode}/runs
POST /scheduler/jobs/{jobCode}/dry-run
  • dry-run 必须返回 willPersist=false、willCallExternal=false、wouldReadCount、wouldProcessCount、riskItems、recommendedActions。
  • dry-run 可以写审计摘要，但不得写 Customer、Order、PlanOrder、Outbox、Exception Queue。

真实执行两步确认

POST /scheduler/jobs/{jobCode}/prepare-run
POST /scheduler/jobs/{jobCode}/confirm-run

  • confirm-run 必须校验确认令牌、权限、任务状态、输入哈希、幂等键和锁。
  • 生产环境 LIVE 执行必须填写执行原因，并记录操作者、角色、数据范围和 traceId。

Outbox 回推

POST /scheduler/outbox/process

  • Outbox 回推失败只更新 Outbox 状态，不回滚本地 PlanOrder finished 或 Order used。

6. 流程

6.1 自动调度锁流程

  1. Scheduler 扫描 ENABLED 且到期的任务。
  2. 通过 MySQL 条件更新抢占锁：lock_until IS NULL OR lock_until < now()。
  3. 抢锁成功写入 lock_owner、run_id、lock_until。
  4. 抢锁失败记录 SKIPPED，不执行任务。
  5. 长任务执行中按 TTL 续租，防止锁过期造成重复执行。
  6. 任务结束后写执行摘要、审计日志、释放锁并计算下一次执行时间。

6.2 dry-run 流程

  1. 读取任务配置、Provider 配置、Mapping 版本、权限和数据范围。
  2. 预估读取数量、处理数量、跳过数量、异常数量和风险项。
  3. 模拟幂等检查、字段映射、状态映射、Customer 匹配、Order 幂等结果。
  4. 返回 willPersist=false、willCallExternal=false、wouldGenerateEvents、wouldReplayThrough。
  5. 仅允许写审计摘要，不落核心业务表、不调用真实外部接口、不入 Outbox、不入异常队列。

6.3 手动真实执行流程

  1. 用户先执行 dry-run，并查看影响范围和风险项。
  2. 用户点击 prepare-run，系统生成确认令牌和执行摘要。
  3. 用户填写执行原因，生产 LIVE 操作输入确认词。
  4. confirm-run 校验确认令牌未过期、输入哈希未变化、权限仍有效、任务未被锁定。
  5. 系统获取 job lock，创建 scheduler_job_run，执行任务。
  6. 执行完成后展示成功数、失败数、跳过数、进入异常数、死信数和下一步建议。

6.4 分销状态回查 / 对账流程

  1. Scheduler 读取外部订单状态或本地 MOCK 对账数据。
  2. 根据 Provider Mapping 转成 distribution.order.paid、distribution.order.cancelled、distribution.order.refund_pending、distribution.order.refunded。
  3. 为每个事件生成稳定 idempotency_key。
  4. 通过统一入站服务可信重放。
  5. paid 事件可创建或匹配 Customer + Order(paid)，但绝不进入 Clue。
  6. cancelled / refund_pending / refunded 只能影响已有 Order 的外部状态，不得创建新订单。
  7. 失败记录进入异常队列，并展示冲突字段、外部订单号、本地订单号、traceId 和处理建议。

6.5 异常队列可信重放流程

  1. 用户查看异常原因、脱敏 raw payload、冲突字段、关联订单、traceId。
  2. 用户先执行异常 dry-run，确认修复配置后会怎样处理。
  3. 用户点击重新入队，系统只读取异常记录原 payload，不允许提交新 payload。
  4. Scheduler 调用统一入站服务重放。
  5. 成功后标记 HANDLED，失败后回到 OPEN 并增加 retry_count。
  6. 超过阈值后给出人工处理建议，禁止静默忽略。

6.6 Outbox 异步回推流程

  1. PlanOrder finished 后，Order 完成状态变化先写 scheduler_outbox_event。
  2. Outbox 事件类型 V1 固定为 crm.order.used。
  3. Scheduler 扫描 PENDING 或到期 FAILED 事件。
  4. 通过 CAS 或事件级锁更新为 PROCESSING。
  5. LIVE 模式生成 timestamp、nonce、idempotency_key 和 signature。
  6. 回推成功标记 SUCCESS。
  7. 回推失败按退避策略进入 FAILED，超过阈值进入 DEAD_LETTER。
  8. Outbox 状态变化不反向修改本地已完成履约状态。

7. 调度安全门禁

门禁规则失败动作
主链路门禁Scheduler 不得直接写 Customer、Order、PlanOrder。阻断执行，记录高风险审计。
分销 paid 门禁distribution.order.paid 可创建 Customer + Order(paid)，但不得进入 Clue。阻断执行并提示映射配置错误。
退款 / 取消门禁退款、取消、退款中事件只允许更新已有订单外部状态。阻断新建订单，进入异常队列。
dry-run 门禁真实执行前必须存在未过期且通过的 dry-run。禁止 confirm-run。
LIVE 门禁LIVE 需要 Provider 已发布、签名配置完整、回调地址有效、Mapping 校验通过。禁止切 LIVE 或执行高风险任务。
锁门禁真实执行必须持有 job lock 或事件级锁。跳过执行并记录 SKIPPED。
幂等门禁无 idempotency_key 不允许创建或重放订单。进入异常队列。
重放门禁异常重放必须绑定异常记录，不能提交任意 payload。拒绝请求并审计。
熔断门禁连续失败、签名异常、重复入站激增、外部超时达到阈值。自动暂停任务，转人工确认。
脱敏门禁raw_data、手机号、token、secret、auth_code、支付流水默认脱敏。拒绝无权限明文查看。

8. 页面体验与验收标准

任务总览

  • 展示业务名称、任务目的、运行模式、启停状态、下一次执行、最近 dry-run、最近真实执行、风险等级。
  • 操作列只放“查看详情”，dry-run、真实执行、停用等动作放入详情抽屉，降低误触风险。

任务详情

  • 用业务语言说明任务用途，例如“把已完成服务的订单回推给外部分销系统”。
  • 展示队列状态、最近执行、最近失败原因、风险限制和下一步建议。

队列状态

  • 按 Outbox、异常队列、死信队列、对账队列、重试队列分 Tab。
  • 展示积压数量、最老等待时间、失败主因、重试次数和处理建议。

异常处理

  • 异常列表中文化展示异常类型、冲突字段、外部订单号、本地订单号、traceId、处理建议。
  • 技术字段默认折叠，payload 和响应内容按权限脱敏。

验收标准

  • 任务列表能看懂每个任务的业务目的，而不是只显示接口名或 cron 表达式。
  • 每个任务都显示运行模式、启停状态、下一次执行、最近 dry-run、最近真实执行结果。
  • 任一真实执行前必须有 dry-run 记录和影响范围，生产 LIVE 必须二次确认。
  • 抖音调度只进入 Clue，不创建 Customer、Order、PlanOrder。
  • 分销状态回查、对账拉取和异常重试必须复用统一入站服务。
  • 分销 paid order 在任何重试、回查、对账场景下都不进入 Clue。
  • cancelled、refund_pending、refunded 不得反向创建 Order。
  • Outbox 回推失败不回滚本地 PlanOrder finished 或 Order used。
  • dead-letter 支持查看、重试、标记处理，并保留完整审计。
  • 连续失败、队列积压、签名失败、幂等冲突可触发暂停或风险提示。
  • 任务配置变更接入配置版本、差异、发布前预览、回滚和审计。
  • raw_data 和敏感字段在日志、dry-run、异常队列中默认脱敏。
  • partner_app 只能访问自身 partner_code 的调度、事件和异常数据。
  • integration_operator 不能修改任务配置，不能绕过确认执行 LIVE 任务。
  • 上线检查未通过时，不能切换到 LIVE 或启动高风险任务。
